                             ## Fix Strategy                                                                                                                                                                     
                             <fix_strategy>                                                                                                                                                                      
                                                                                                                                                                                                                 
                             1. Root Cause and Impact Analysis                                                                                                                                                   
                                - The vulnerability arises because `Expander.expand(...)` constructs extraction paths by naively resolving entry names against the target directory (e.g.                        
                             `destDir.toPath().resolve(entry.getName())`) without canonicalizing (normalizing) out .. segments or absolute-name injections.  As a result, a crafted entry                    
                             name like `q/r/../../evil` or `/etc/passwd` bypasses the simple `startsWith` checks and permits files to be written outside the intended extraction directory.                      
                                - The default implementation of `ArchiveEntry.resolveIn(Path parentPath)` similarly does a plain `parentPath.resolve(name)` and only then checks `startsWith`,                   
                             so any unresolved .. segments fool the check.                                                                                                                                     
                                                                                                                                                                                                                 
                             2. High-Level Fix                                                                                                                                                                   
                                A. Canonicalize (normalize) both the parent directory and the resolved output path before enforcing the no-escape check.                                                       
                                B. Centralize all entry-to-file resolution logic through the fixed `ArchiveEntry.resolveIn(Path)` API, and update the Expander example to use it consistently                    
                             instead of ad-hoc path concatenation.                                                                                                                                               
                                                                                                                                                                                                                 
                             3. Detailed Patch Steps                                                                                                                                                             
                                1. Modify `ArchiveEntry.resolveIn(Path parentPath)` in                                                                                                                           
                                   `src/main/java/org/apache/commons/compress/archivers/ArchiveEntry.java`:                                                                                                      
                                   a. At the top, obtain a normalized absolute parent path:                                                                                                                      
                                      ```                                                                                                                                                                        
                                      Path normalizedParent = parentPath.toAbsolutePath().normalize();                                                                                                           
                                      ```                                                                                                                                                                        
                                   b. Resolve and normalize the target path:                                                                                                                                     
                                      ```                                                                                                                                                                        
                                      Path resolved = normalizedParent.resolve(getName()).normalize();                                                                                                           
                                      ```                                                                                                                                                                        
                                   c. Enforce the traversal check against the normalized parent:                                                                                                                 
                                      ```                                                                                                                                                                        
                                      if (!resolved.startsWith(normalizedParent)) {                                                                                                                              
                                          throw new IOException(String.format(                                                                                                                                   
                                              "Zip slip '%s' + '%s' -> '%s'",                                                                                                                                    
                                              normalizedParent, getName(), resolved));                                                                                                                           
                                      }                                                                                                                                                                          
                                      ```                                                                                                                                                                        
                                   d. Return `resolved` (not the raw `parentPath.resolve(name)`).                                                                                                                
                                2. In the Expander example (`src/main/java/org/apache/commons/compress/archivers/examples/Expander.java`),                                                                       
                                   for every overload of `expand(...)` (around lines 86, 204, 332, 406, 440):                                                                                                    
                                   a. Replace any ad-hoc path construction (e.g.                                                                                                                                 
                                      `Path outFile = destDir.toPath().resolve(entry.getName());`)                                                                                                               
                                      with a call to the secure API:                                                                                                                                             
                                      ```                                                                                                                                                                        
                                      Path outFile = entry.resolveIn(destDir.toPath());                                                                                                                          
                                      ```                                                                                                                                                                        
                                   b. Leave downstream logic (directory creation, `Files.copy`, permission setting, etc.) unchanged.                                                                             
                                3. Re-run existing unit tests and example runs to verify no regressions in normal (non-traversal) extractions.                                                                   
                                                                                                                                                                                                                 
                             4. Justification                                                                                                                                                                    
                                - Normalizing both the parent directory and the resolved path ensures all .. segments and absolute-path injections are collapsed before performing the                         
                             `startsWith` check, effectively blocking path traversal.                                                                                                                            
                                - Centralizing resolution through `ArchiveEntry.resolveIn(...)` keeps the fix localized (no need to sprinkle normalization logic in multiple code sites) and                     
                             preserves existing functionality for benign entry names.                                                                                                                            
                                - This approach aligns with the common Zip Slip mitigation pattern (using `File.getCanonicalPath()` or `Path.normalize()`), and it does not modify any fuzzing                 
                             harness, tests, or utility code outside the core extraction logic.                                                                                                                  
                                - Side-effects for valid extraction paths are minimal: the only difference is that the returned `Path` is already normalized, which matches the expected user                    
                             view of file locations and does not interfere with directory creation or content copying.                                                                                           
                                                                                                                                                                                                                 
                             5. Mitigation of Potential Side-Effects                                                                                                                                             
                                - If consumers rely on the non-normalized path for metadata (unlikely in an example tool), they will now get the canonical path, which is more semantically

                             common Zip Slip vector demonstrated by the fuzzer.

                             Patch Attempt                                                                                                                                                                       
                             - In `ArchiveEntry.resolveIn(Path parentPath)`, the patch adds:                                                                                                                     
                               - `normalizedParent = parentPath.toAbsolutePath().normalize()`                                                                                                                    
                               - `resolved = normalizedParent.resolve(name).normalize()`                                                                                                                         
                               - Checks that `resolved.startsWith(normalizedParent)` and throws an `IOException` if not.                                                                                         
                               - Returns the normalized `resolved` path instead of the unguarded `outputFile`.

                             preliminary call to `entry.resolveIn(destDir.toPath())`, assigning the result to a local `outFile` (but not actually using it to open or write streams).                            
                             - The patch text (as shown) interleaves those `Path outFile = entry.resolveIn(destDir.toPath());` lines in many methods (line numbers ~83, 201, 329, 403, 437,
